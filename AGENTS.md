# bootstrap-material-design

## Role
This repo contains shared Bootstrap 3 Material Design theme assets used by the WebStarts admin/editor UI. It is a styling/theme repo, not the main builder app and not customer-site content.
It is effectively a Bootstrap makeover layer: Bootstrap 3 structure and components are restyled to look and behave closer to Material Design, with WebStarts-specific editor/dashboard additions layered on top.

## Git
- Repo root: `~/projects/webstarts/bootstrap-material-design`
- Current branch: `master`

## Tech Stack
- Bootstrap 3 theme assets
- Sass/SCSS and JavaScript
- Grunt-era build tooling

## Key Areas
- `sass/`: source Sass partials
- `sass/editor-v3.scss`: editor v3 stylesheet entry (autogenerated; edit imported partials)
- `sass/editor/`: editor-specific Sass
- `sass/editor/_editor.scss`: main editor shell/chrome styling
- `sass/editor/_toolbox.scss`: editor toolbox controls
- `sass/editor/_widgets.scss`: editor widget previews/config UI
- `sass/editor/_file-manager.scss`: file manager UI
- `sass/editor/_sidebar.scss`: editor sidebar
- `sass/_wsui.scss`: WebStarts UI helpers and local utility patterns
- `sass/pages/dashboard/_index.scss`: dashboard/site thumbnail styles
- `sass/bootstrap-material-design.scss`: main Bootstrap Material entry (autogenerated; edit imported partials)
- `sass/bootstrap-material-design-designer.scss`: designer variant entry (autogenerated; edit imported partials)
- `less/`: legacy Less sources; the build pipeline converts Less to Sass so the canonical checked-in Sass under `sass/` stays in sync. Prefer editing the source that owns the change (often `less/` partials that feed generation)
- `scripts/`: JavaScript plugin/theme scripts
- `dist/`: generated distribution assets, when present

## Working Rules
- Always edit the source partial (Sass or Less) that owns the change; do not edit generated files.
- Preserve class naming conventions and utility patterns for compatibility with admin/editor UI.
- For new UI patterns, check for existing equivalents before adding new classes.
- Run `grunt` to build assets after changes.

## Testing & Validation
- Build assets: `grunt`
- Validate output: check `dist/` for updated CSS/JS
- Test in WebStarts admin/editor UI for visual/functional correctness

## Cross-Repo Integration

The main WebStarts admin/editor app lives in `../webstarts/`. Its HTML/CSS
leans heavily on class patterns that originate in this repo—not only the
namespaced helpers below but also Bootstrap-style layout and utilities such as
`d-flex`, `align-items-center`, `justify-content-center`, `w-100`, grid columns
(`col-*`, `col-sm-*`, etc.), and spacing utilities (`m-*`, `p-*`, and related
margin/padding scales). When you add or change WebStarts UI HTML under
`webstarts/`, reuse these established classes so screens stay consistent with
the shared theme. **Avoid** introducing new `<style>` tags or inline
`style="..."` for things already expressible with these classes; if a pattern
repeats, add or extend a partial here (e.g. `_wsui.scss`) instead.

Agents should start from [../webstarts/AGENTS.md](../webstarts/AGENTS.md), which
links here and defines the full read order for WebStarts UI work.

- `.wsui`: wrapper namespace for WebStarts admin/editor UI helpers. It provides
  box sizing, form/input helper patterns, toggle styles, icon spacing, upload
  buttons, cutout buttons, layout helpers, and compatibility utilities.
- `.wsui.wsme`: WebStarts main editor shell. It styles the editor layout,
  header/menubar, sidebar interaction, edit area, iframe frame, and phone
  viewport presentation.
- `.editor-toolbox`: toolbox panels and controls used by the v3 editor, such as
  background image previews, 9-way image alignment, option pickers, small FABs,
  and slider/input group styling.
- `.dashboard-*`: dashboard-specific layout and site-thumbnail styles.
- `.material-icons`: Material icon font styling used throughout buttons,
  nav/sidebar items, dropdowns, cards, toolbox controls, and dashboard UI.
- Bootstrap/MDB classes: `.btn`, `.btn-fab`, `.btn-raised`, `.form-group`,
  `.checkbox-material`, `.radio`, `.dropdown-menu`, `.card`, `.progress-material`,
  and plugin classes are still Bootstrap-shaped, with Material-like styling.
- Utility/layout classes: prefer existing helpers such as `.d-flex`,
  `.align-items-center`, `.justify-content-center`, `.w-100`, `.m-*`, `.p-*`,
  `.col-*`, and `.col-sm-*` before adding new layout CSS.
- Text and box helpers: keep standard heading/body patterns such as `h2`/`.h2`
  and global `border-box` sizing in mind when adjusting spacing or component
  proportions.
- Popup classes: reuse existing modal/popup patterns before inventing a new
  overlay. WebStarts admin code commonly creates popups through `WSUI.popup`
  and related classes in `webstarts/cadmin/scripts/js`.

Longer-term direction for WebStarts UI is a component-based setup where each
feature ships its own CSS, JavaScript, and HTML together, instead of growing only
through global theme partials and scattered markup. Until that migration lands,
treat this repo as the source of truth for shared class families and generated
Sass.

- If the task is adding/changing WebStarts editor UI markup in `webstarts/`,
  inspect this repo first for existing class patterns before inventing a new
  component style.
- Customer-site runtime CSS is owned by `../webstarts/`; see
  [../webstarts/AGENTS.md](../webstarts/AGENTS.md).
- `webstarts/cadmin/jslib/src/v3/static_css.php` controls which editor CSS is

  screen being changed.
- `designer/clients/cadmin` in `webstarts/` is a symlink to `cadmin/`; UI
  changes may be seen through both paths.

## Working Rules

- Treat this as shared UI infrastructure. Small visual changes can affect many
  admin/editor screens.
- Prefer editing source Sass over generated CSS when a source exists.
- Do not edit top-level generated Sass entry files just to add styles. Put new
  rules in the relevant partial, such as `_wsui.scss` or `sass/editor/*.scss`.
- Match existing Bootstrap 3 and local Sass conventions; this is not Bootstrap
  4/5 and not a modern design system.
- Prefer existing class families over new one-off classes: `.wsui` helpers for
  admin chrome, `.editor-toolbox` for editor controls, `.dashboard-*` for
  dashboard screens, and Bootstrap/MDB button/form classes for common controls.
- Keep Material-style polish consistent: subtle shadows, restrained radii,
  Material Icons where already used, predictable hover/focus states, and dense
  admin/editor layouts.
- Avoid broad theme refactors unless explicitly requested.
