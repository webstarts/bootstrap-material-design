# bootstrap-material-design

## Role

This repo contains shared Bootstrap 3 Material Design theme assets used by the
WebStarts admin/editor UI. It is a styling/theme repo, not the main builder app
and not customer-site content.

It is effectively a Bootstrap makeover layer: Bootstrap 3 structure and
components are restyled to look and behave closer to Material Design, with
WebStarts-specific editor/dashboard additions layered on top.

## WebStarts UI Class Families

The main WebStarts admin/editor app lives in `../webstarts/`. Its HTML/CSS
leans heavily on class patterns that originate in this repo—not only the
namespaced helpers below but also Bootstrap-style layout and utilities such as
`d-flex`, grid columns (`col-*`, `col-sm-*`, etc.), and spacing utilities (`m-*`,
`p-*`, and related margin/padding scales). When you add or change WebStarts UI
HTML under `webstarts/`, reuse these established classes so screens stay
consistent with the shared theme. **Avoid** introducing new `<style>` tags or
inline `style="..."` for things already expressible with these classes; if a
pattern repeats, add or extend a partial here (e.g. `_wsui.scss`) instead.

Agents should start from [../webstarts/AGENTS.md](../webstarts/AGENTS.md), which
links here and defines the full read order for WebStarts UI work.

- `.wsui`: wrapper namespace for WebStarts admin/editor UI helpers. It provides
  box sizing, form/input helper patterns, toggle styles, icon spacing, upload
  buttons, cutout buttons, layout helpers, and compatibility utilities.
- `.wsui.wsme`: WebStarts main editor shell. It styles the editor layout,
  header/menubar, sidebar interaction, edit area, iframe frame, and phone
  viewport presentation.
- `.editor-toolbox`: toolbox panels and controls used by the v3 editor, such as
  background image previews, 9-way image alignment, option pickers, small FABs,
  and slider/input group styling.
- `.dashboard-*`: dashboard-specific layout and site-thumbnail styles.
- `.material-icons`: Material icon font styling used throughout buttons,
  nav/sidebar items, dropdowns, cards, toolbox controls, and dashboard UI.
- Bootstrap/MDB classes: `.btn`, `.btn-fab`, `.btn-raised`, `.form-group`,
  `.checkbox-material`, `.radio`, `.dropdown-menu`, `.card`, `.progress-material`,
  and plugin classes are still Bootstrap-shaped, with Material-like styling.

Longer-term direction for WebStarts UI is a component-based setup where each
feature ships its own CSS, JavaScript, and HTML together, instead of growing only
through global theme partials and scattered markup. Until that migration lands,
treat this repo as the source of truth for shared class families and generated
Sass.

## Main-App Relationship

- The main app is `../webstarts/`; read
  [../webstarts/AGENTS.md](../webstarts/AGENTS.md) before changing UI that is
  wired into editor/admin behavior.
- For **how** admin HTML is built in jQuery (e.g. `WSUI.popup`, grid/flex
  classes in `cadmin/scripts/js`), read
  [../webstarts/cadmin/AGENTS.md](../webstarts/cadmin/AGENTS.md).
- If the task is adding/changing WebStarts editor UI markup in `webstarts/`,
  inspect this repo first for existing class patterns before inventing a new
  component style.
- Customer-site runtime CSS is owned by `../webstarts/`; see
  [../webstarts/AGENTS.md](../webstarts/AGENTS.md).
- `webstarts/cadmin/jslib/src/v3/static_css.php` controls which editor CSS is
  loaded by the main app. Confirm the target Sass/CSS actually reaches the
  screen being changed.
- `designer/clients/cadmin` in `webstarts/` is a symlink to `cadmin/`; UI
  changes may be seen through both paths.

## Working Rules

- Treat this as shared UI infrastructure. Small visual changes can affect many
  admin/editor screens.
- Prefer editing source Sass over generated CSS when a source exists.
- Do not edit top-level generated Sass entry files just to add styles. Put new
  rules in the relevant partial, such as `_wsui.scss` or `sass/editor/*.scss`.
- Match existing Bootstrap 3 and local Sass conventions; this is not Bootstrap
  4/5 and not a modern design system.
- Prefer existing class families over new one-off classes: `.wsui` helpers for
  admin chrome, `.editor-toolbox` for editor controls, `.dashboard-*` for
  dashboard screens, and Bootstrap/MDB button/form classes for common controls.
- Keep Material-style polish consistent: subtle shadows, restrained radii,
  Material Icons where already used, predictable hover/focus states, and dense
  admin/editor layouts.
- Avoid broad theme refactors unless explicitly requested.
